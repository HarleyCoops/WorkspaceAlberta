# Deployment egress allowlist — required hosts

> Captured from a Claude Code on the web session on 2026-06-20. The remote
> container could not reach the deployment because the two hosts below are not
> on the environment's network egress allowlist. This file records the exact
> diagnosis and fix so it survives the ephemeral container.

## Symptom

Every request to either endpoint returns:

```
HTTP/2 403
x-deny-reason: host_not_allowed
content-type: text/plain

Host not in allowlist: <host>. Add this host to your network egress settings to allow access.
```

This affects all clients inside the container equally — `curl`, the MCP
client, and the agent — because they all share the same egress path.

## Diagnosis (verified)

The `403` is **not** coming from the Cloud Run service. It is a synthetic
response generated by this environment's egress proxy
(`local_proxy@127.0.0.1`). Evidence:

- The `x-deny-reason: host_not_allowed` header and the
  "Add this host to your network egress settings" body are proxy-generated,
  not application output.
- DNS "resolves" in ~4 ms and TCP "connects" in ~5 ms because the request only
  reaches the *local* proxy; it is intercepted and answered there and never
  leaves the box for the real server.

So the deployment itself may be healthy — it simply cannot be reached from a
container whose egress allowlist excludes these hosts. There is **no in-repo
configuration** that controls this; the allowlist is a managed
environment-level setting.

## Required hosts

```
elbowsupknivesout.warreandvavasour.com
workspacealberta-719334491060.northamerica-northeast1.run.app
```

## Fix

1. Open the Claude Code web app → this environment's settings →
   **Network / egress access**.
2. Set a **custom allowlist** (or edit the existing one) and add both hosts
   above.
3. Save, then start a **fresh session** — egress policy is applied at
   container start.
4. Re-test:

   ```bash
   curl -sS -D - https://workspacealberta-719334491060.northamerica-northeast1.run.app
   curl -sS -D - https://elbowsupknivesout.warreandvavasour.com
   ```

   Any response other than the proxy's `host_not_allowed` 403 means egress is
   open and the real server is now reachable. From there, any remaining issue
   is an application/deployment concern that can be debugged normally.

Docs: https://code.claude.com/docs/en/claude-code-on-the-web (network-access /
egress policy section).
